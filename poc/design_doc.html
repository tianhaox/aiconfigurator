AIC Perf Engine — Python + Rust Split

Design doc · greenfield PoC validated locally · June 2026
TL;DR

Split AIC's perf engine into two layers: Python owns model building (HF parsing, op-list composition, quant resolution), Rust owns execution (compiled artifact + per-op latency math, hot path). Both Python sweep callers and external Rust callers (e.g. Dynamo Mocker) use the same Rust core function via two binding paths (PyO3 / direct crate use).

The model-building knowledge stays in Python (one source of truth), but the hot path runs in Rust at GIL-free, rayon-friendly speed.

A throwaway PoC under poc/ validates the architecture end-to-end: 14/14 parity tests pass bit-identical against a pure-Python reference; an external Rust binary loads the same artifact and runs multi-threaded.

1. Motivation

1.1 Current state

AIConfigurator predicts LLM serving performance for many (parallel config, batch_size, ctx_tokens, quant_mode, ...) combinations under SLA targets. Today the entire pipeline is Python:

Task spec → sweep loop → Predictor strategy → backend.run_static ↓ for op in model.context_ops: op.query(database, ...) ↓ sum, scale, return

Two Rust hooks already exist:

  • rust/aiconfigurator-core/ — a ctypes-backed sidecar for Dynamo Mocker. Receives a model identity via JSON; internally re-derives the op list from name + parallel sizes.
  • sdk/rust_engine_step.py — Python wrapper for the above.

1.2 Pain points

Pain pointCost today
Python GIL serializes sweep evaluation Sweep with ~2,250 points takes ~0.6s; mostly DB look-ups + DataFrame ops on a single core.
Model knowledge duplicated in Python and Rust aiconfigurator-core/model.rs (~350 lines) re-derives the op list. Every new Python model needs a parallel Rust update.
External Rust callers (Mocker) must round-trip through Python for build JSON ↔ struct conversion every per-point call; coupled to Python interpreter lifecycle.
DataFrame-heavy hot loop pandas/concat ops dominate wallclock at low op counts; type drift bugs (object vs int64) caught only at parity.

2. Proposal: Python + Rust Split

2.1 Three-layer architecture

The new design separates what to compute (Python's job — model topology, quant semantics, layer counts) from how fast / how multi-threaded to compute it (Rust's job). Between them sits a bytes-encoded compiled artifact — a plain-data representation of the op list. The bytes are the architectural boundary; disk files, in-memory pipes / stdin, shared memory, and include_bytes! are all valid transports for those bytes.

Python Rust ───────────────────────────── ───────────────────────────── parse HF config (model arch) ↓ instantiate Operation objects (one per layer-level op) ↓ produce list[OpSpec] ─────── compile/build ─────────► build Engine: Vec + metadata │ engine.to_bytes() ──── bytes ─────► Engine::from_bytes(&buf) (in-memory; or save_bin (in-memory pipe / disk / stdin — for disk transport) same bytes, alternate packaging) │ ↓ for (batch, seq, mode) in sweep: run_static(engine, db, batch, seq, mode) call into Engine ────────► for op in engine.op_list: lookup db, scale, accumulate return HashMap<op_name, latency> ◄────── result ──────────────────

Two real-world examples of the same architectural pattern (Python builds → bytes/JSON cross → Rust deserializes → Rust runtime):

PatternEncodingExample
Dynamo MockEngineArgs JSON string MockEngineArgs.from_json(json.dumps(raw)) (replay/main.py:180)
PoC Engine artifact bincode bytes engine.to_bytes() / engine_from_bytes(blob)

2.2 The Engine abstraction

The new central object is the Engine: a Rust struct holding a compiled op list plus metadata. Its sole runtime method is run_static(db, batch, seq, mode) → HashMap<name, latency>.

PropertyEngineV1 InferenceSession (deleted)V2 stateless Predictor (current)
State heldCompiled op list + metadatamodel + db + backendNone (everything passed per call)
Hot path locationRustPythonPython (delegates to backend)
Bound to one model?Yes (the compiled one)YesNo
Sweep methods?No (sweep lives in Python sweep.py)Yes (find_best_*)No
Callable from Rust?Yes (Mocker, future tools)NoNo

The Engine is conceptually the execution unit that V1's InferenceSession tried to be. V2 went stateless as an overcorrection against V1's three-things-in-one design; the PoC shows that strategy (stateless Predictor) and execution unit (stateful Engine) are different layers and both belong.

2.3 Two callers, one Rust core

┌──────────────────────────────────┐ │ Engine::run_static │ │ (rust/aic-step/src/engine.rs) │ ◄── single source of truth │ pure Rust, zero Python │ └──────────────▲───────────────────┘ │ ┌────────────────┴───────────────┐ │ │ ┌─────────┴───────────┐ ┌──────────┴──────────┐ │ PyO3 binding │ │ Rust crate direct │ │ Engine.run_static │ │ engine.run_static() │ │ (PyEngine wrap) │ │ (from bytes/disk) │ └─────────┬───────────┘ └──────────┬──────────┘ │ │ Python: Task / sweep External Rust callers (releases GIL on call) (Mocker, future tools)

Both bindings hit the same run_static. The PyO3 wrapper releases the GIL during execution, so Python-side multithreading (or batch entry points) parallelizes without the GIL bottleneck.

3.2 End-to-end flow

Python path

from aic_poc import MockLLMModel
from aic_step import build_engine, DbHandle

model = MockLLMModel(hidden=4096, n_layers=32)  # Python: build op list

engine = build_engine(model)                     # PyO3: Python → Rust
db = DbHandle.load("data/gemm_perf.parquet")     # arrow-rs: load DB

# Hot path: PyO3 wrapper releases GIL, executes pure-Rust core,
# returns Python dict[str, float].
result = engine.run_static(db, batch_size=8, seq_len=2048, mode="static")

External Rust caller (mocker_demo)

use aic_step::{Database, Engine, StaticMode};

// Primary mode: receive Engine bytes over stdin / pipe / shared mem.
let mut buf = Vec::new();
std::io::stdin().read_to_end(&mut buf)?;
let engine = Arc::new(Engine::from_bytes(&buf)?);
// (or, alternate packaging) let engine = Engine::load_bin("compiled.bin")?;
let db = Arc::new(Database::load_gemm_parquet("data/gemm_perf.parquet")?);

// Same hot path as Python; no PyO3 in the picture.
let result = engine.run_static(&db, 8, 2048, StaticMode::Full)?;

// Trivial rayon fan-out across many points:
points.par_iter().for_each(|p| {
    let _ = engine.run_static(&db, p.batch, p.seq, StaticMode::Full);
});

3.3 Build / run

# one-time
brew install rust
python3.12 -m venv .venv && source .venv/bin/activate
pip install maturin pyarrow pytest

# build PyO3 extension into the venv
cd poc/
maturin develop --release

# generate test data and run parity tests
python data/build_gemm_parquet.py
pytest                                          # 14/14 pass

# build the pure-Rust binary (no libpython)
cargo build --release --bin mocker_demo --no-default-features
./target/release/mocker_demo compiled.bin data/gemm_perf.parquet
Design Note

pyo3 is an optional dependency behind a python Cargo feature. With it, the lib outputs a Python extension (aic_step._native.so). Without it (the binary build), the lib is a plain Rust crate and the binary links zero Python symbols. This is the cleanest way to dual-purpose one crate for both callers.

Bit-Identical

Every parameterization of the parity test asserts math.isclose(rust_val, py_val, rel_tol=1e-12, abs_tol=1e-12) per op. All 27 (op × test) checks passed on first build. The arithmetic is simple enough that Rust f64 and Python float match exactly.

5. Open Decisions

5.1 Naming

Picked Engine (over CompiledModel / InferenceSession / PerfModel) because:

  • Aligns with existing rust_engine_step.py in the repo.
  • TensorRT/vLLM call compiled artifacts "engine" — pattern match.
  • Avoids resurrecting InferenceSession, which V2 just deleted, and which falsely implies "running inference" (we predict perf, we don't run the model).
  • Short. Clean. Engine.run_static(...) reads well.

If "engine" ever conflicts with real LLM-runtime "engine" in shared docs, escalate to PerfEngine; rename cost is trivial.

5.2 Predictor vs Engine

The current Phase 1 SDK has a Predictor Protocol (stateless strategy). The Engine is a different layer — stateful execution unit. Both belong:

LayerStatefulnessRoleExamples
PredictorstatelessHow to evaluate one point AnalyticPredictor, MockerPredictor (future)
Enginestateful (compiled artifact)Concrete execution unit Engine (Rust), wrapped by EnginePredictor

The integration path: an EnginePredictor implements the Predictor Protocol; internally it caches Engines per (model_id, parallel_config). Caller-side API is unchanged.

5.3 Artifact format

PoC uses bincode for the Engine serialization (fast, compact, Rust-native). Considered Arrow IPC and JSON; bincode wins on size/speed. JSON debugging is achievable via a dump tool if needed.

5.4 DB layout

PoC's DB is per-op (GEMM table + per-GPU GpuSpec for analytic ops). Production DB has ~10 op kinds (attention, MoE, comm, MLA, etc.). The PoC's table abstraction (Database { gemm: GemmTable, gpu: GpuSpec, ... }) maps each op kind to its own Rust struct, matching today's "op owns its data" Python layout. Extending: one new struct per op, ~50–200 lines each.

5.5 Op kinds in the PoC

  • Gemm — table-lookup cost model (db[(m,n,k)] * batch*seq * scale_factor).
  • Dsa — DeepSeek-style sparse attention; latency = max(compute_roofline, memory_roofline) × n_layers (SoL — needs only GpuSpec, no per-shape DB row).

6. Roadmap (if we proceed)

  1. Decide: Adopt the Engine architecture? Or keep PoC on the shelf? Inputs to the decision: the validation results above, plus a stress bench (next item).
  2. Stress bench: extend the PoC to N=10k points sequential vs rayon. Measure real wallclock speedup. (Expected: near-linear up to core count.)
  3. Op coverage: incrementally port the remaining ~9 op kinds (attention, MoE, comm, MLA, custom_allreduce, ...). One PR per op; parity test per op.
  4. DB completeness: extend DbHandle to cover non-GEMM tables; interpolation/extrapolation logic moves to Rust (current Python perf_database.py arithmetic ports cleanly).
  5. Integration into Task.run(): introduce an EnginePredictor that the existing Phase 1 Predictor Protocol can return; gate behind a Task field for opt-in (parity tests both paths).
  6. Production wheels: maturin CI for macOS / Linux / Windows × Python 3.10–3.12. uv add aic_step-friendly.

Estimated effort (single dev with LLM assist): items 2–3 are ~3–4 weeks; item 4 is ~2 weeks; item 5 is ~1 week; item 6 is ~1 week.

7. Appendix: Naming Notes

For internal sanity, the four words floating in this design:

WordMeaning hereWhat it is NOT
Predictor Strategy (stateless interface) — Phase 1 V2 abstraction The thing that holds the compiled model
Engine Execution unit — holds compiled op list, runs run_static An LLM runtime; the predictor strategy itself
compile_model() / build_engine() Marshal Python Operation objects into a Rust Engine (PyO3 type conversion + struct construction) "Compile" in the LLVM/JIT sense — there's no codegen
OpSpec Plain-data Rust representation of one op (enum variant + params) An Operation Python class — that lives in Python
